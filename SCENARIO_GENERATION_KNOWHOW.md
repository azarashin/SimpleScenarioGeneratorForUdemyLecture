# シナリオ・画像生成の改善ノウハウ

## 今回、分量が改善した理由

今回の改善で最も効果があったのは、本文全体の文字数を増やす指示ではなく、
**セリフを独立した会話ターンとして生成し、その件数を明示・検証したこと**です。

以前の生成条件は次のようなものでした。

- 本文全体を800〜1600文字にする
- ナレーションとセリフをそれぞれ1件以上含める

この条件では、モデルは長いナレーションや少数の長い発言だけでも要件を満たせます。
そのため、セリフが10〜20件程度、場合によってはそれ以下に収束していました。

改善後は次の条件を組み合わせています。

- 1セクションに20〜40件の `dialogue` ブロックを要求する
- 1回の発言を1つの `dialogue` ブロックとして分割する
- 複数人が登場する場合は、話者を自然に交代させる
- 会話を導入、疑問・緊張、応答、補足・反応、状況の進展という複数のビートで構成する
- 生成後にセリフ件数を機械的に検証する
- 条件不足の出力を採用せず、実測値を添えて再生成する

プロンプトによる指示だけでなく、品質検証でも件数を保証している点が重要です。
モデルが指示を外した場合でも、少ないセリフの成果物がそのまま採用されません。

## 推奨設定

現在の推奨値は次のとおりです。

```json
{
  "scenario_body_generation": {
    "subsections_per_section": 3,
    "target_characters": 1200,
    "min_characters": 850,
    "max_characters": 1600,
    "min_dialogue_blocks": 6,
    "max_dialogue_blocks": 14,
    "require_event_mentions": true
  }
}
```

文字数は空白を除いて計測します。セリフ数は `type=dialogue` のブロック数です。

## サブセクション数とアウトラインレビューの規模

### `subsections_per_section`は本文量だけでなくアウトラインの大きさも増やす

`subsections_per_section`は、1節を何回の本文生成に分割するかを指定します。この値を増やすと
本文量だけでなく、Step 02で設計・レビューするイベント、開始状態、終了状態、状態差分も
同じ比率で増えます。

例えば、5章、各章6節、各節12サブセクションの場合は次の規模になります。

```text
5章 × 6節 × 12サブセクション = 360サブセクション
```

各サブセクションが850〜1,600文字の場合、本文全体は空白を除いて約30.6万〜57.6万文字です。
さらにアウトラインレビューでは、360件それぞれについてイベント、開始状態、不可逆な変化、
終了状態、伏線、人物変化などを構造化JSONで返す必要があります。

この規模を1回のモデル応答で全面的に修正させると、実際に次の失敗が発生しました。

- 後半の章が出力されず、5章必要なところ2章だけになる
- 章番号がすべて`1`になる
- 節やサブセクションが欠落する
- 汎用プレースホルダーが具体的な出来事へ置き換わらない
- 空の伏線・人物アーク台帳が残る
- 同じ巨大な生成をリトライし、API利用量と待ち時間だけが増える

このため、通常は`subsections_per_section: 3`を推奨します。

```json
{
  "scenario_body_generation": {
    "subsections_per_section": 3,
    "target_characters": 1200,
    "min_characters": 850,
    "max_characters": 1600
  }
}
```

この設定では1節が約2,550〜4,800文字、生成目標では約3,600文字になります。さらに長くする場合も、まず`4`または`5`で
1章分を試し、アウトライン成果物、API利用量、本文の反復を確認します。分量だけを増やしたい場合、
サブセクション数を大幅に増やす前に`target_characters`や`max_characters`の調整も検討します。

### 全章の整合性レビューは全体設計と章単位の具体化へ分割する

全章の整合性は確認する必要がありますが、全章の詳細なアウトラインを1回の応答で再出力させる
必要はありません。現在のStep 02 Reviewは次の単位へ分割しています。

```text
全章のイベント配置
    ↓
物語全体の台帳を設計
  - プロットスレッド
  - 伏線の提示と回収
  - キャラクターアーク
    ↓
第1章を具体化
    ↓
第2章を具体化（第1章の結果を参照）
    ↓
以降、章単位で具体化
```

全体台帳では「どのイベントIDで開始・提示・転換・解決・回収するか」だけを先に決めます。
章単位レビューでは、その台帳を契約として各節・サブセクションを具体化します。これにより、
全章の整合性を維持しながら、1回の出力サイズを抑えられます。

章番号、節番号、サブセクション番号、イベントIDはモデルに再採番させず、元のアウトラインを
正としてコード側でも復元します。一方、節数やイベント数そのものが欠落した場合は、欠落した章を
示して失敗させます。番号だけを直して不完全なアウトラインを採用しないことが重要です。

### 正常な分割呼び出しとリトライをログで区別する

章単位レビューでは、全体台帳1回と章数分のAPI呼び出しが正常時にも発生します。これはリトライ
ではありません。`trace.jsonl`では次のイベントで確認できます。

- `outline_review_unit_started`: 台帳または章のレビュー開始
- `outline_review_unit_succeeded`: そのレビュー単位の成功
- `step_retry_scheduled`: Step 02 Review全体の再試行

例えば5章なら、正常時は台帳1件と章5件の成功イベントが記録されます。同じ章へ進む前に
`step_retry_scheduled`が記録されている場合は、分割処理ではなく実際のリトライです。

章単位の検証に失敗した場合は、その章だけを再試行します。例えば「第3章が1節しか返さず、
6節必要」という失敗理由を次の第3章プロンプトへそのまま追加し、完全な章を再出力させます。
成功済みの全体台帳と章は`artifacts/outline-review-units/`へ保存され、同じ入力での再実行時に
再利用されます。章単位の再試行を使い切った後は、同じ失敗をStep全体の再試行で繰り返しません。

章単位では正しく見えても、複数章の結合後に同じスレッドを二度解決したり、台帳とは異なる
イベントで伏線を提示したりする場合があります。プロットスレッドの開始・解決、伏線の提示・回収、
人物アークの転換位置は、章出力より全体台帳を優先し、結合時に台帳のイベントIDへ機械的に
再配置します。これにより全章レビューのためだけに成功済み章を再生成する必要がなくなります。

- `outline_review_unit_failed`: 対象台帳・章、単位内試行番号、具体的な失敗理由
- `outline_review_unit_checkpoint_loaded`: 成功済み単位をAPIなしで再利用

以前の一括レビューでは、1試行に1〜3分かかった後、同じ規模の処理を最大5回繰り返す事例が
ありました。調査中は失敗原因を素早く確認するため、検証用設定でリトライを減らす方法もあります。

```json
{
  "retry_strategy": {
    "short_retries": 0,
    "prompt_revision_retries": 0,
    "fallback_enabled": false
  }
}
```

原因を解消した後は通常設定へ戻します。リトライ回数を増やしても、出力上限を超える一括レビューや
構造的に大きすぎるアウトラインは改善しません。先に生成単位を小さくすることが必要です。

### 失敗後の再実行

サブセクション数またはアウトライン設計を変更した場合は、Step 02から再生成します。

```powershell
python run_pipeline.py `
  --config examples/pipeline.openai.config.json `
  --input examples/rough-idea.txt `
  --run-id rough-idea-scenario-001 `
  --from-step step-02-generate-outline
```

`--from-step`でStep 02自体が再実行されるため、この用途では通常`--force`は不要です。

## 文字数とセリフ数の役割

文字数とセリフ数は、異なる品質を制御します。

- `min_dialogue_blocks` は会話の往復回数を保証する
- `max_dialogue_blocks` は細切れになりすぎることを防ぐ
- `min_characters` は内容が極端に薄い出力を防ぐ
- `max_characters` は1セクションが長くなりすぎることを防ぐ

今回の目的ではセリフ数が主要な指標です。実際に23セリフ、5ナレーション、
1331文字のセクションは十分な会話量を持っていましたが、文字数下限を1400にすると
不必要に拒否されました。この知見を踏まえつつ、現在は完成コンテンツとして十分な
読み応えと生成安定性を両立するため、1セクションを既定で3サブセクションに分けます。
各サブセクションは1,200文字を生成目標、850〜1,600文字を合格範囲とし、生成後に
1つのセクションへ結合します。分量を変更する用途では、分割数、生成目標、合格範囲を
セットで調整します。

文字数下限を高くしすぎると、次の問題が起きます。

- 条件を満たすためだけの冗長な説明が増える
- 長い独白やナレーションで文字数を稼ぎやすくなる
- 十分な会話量がある自然なセクションまで拒否される
- 再試行回数とAPI利用量が増える

実運用では、下限1,000文字に対して補完後981文字となり、内容と13件のセリフを持つ出力が
19文字不足だけで4試行すべて不合格になる事例がありました。この程度の境界差は品質差より
モデル出力の揺らぎの影響が大きいため、生成目標1,200文字を維持したまま合格下限を850文字へ
緩和しています。目標値を下げず、合格下限だけに余裕を持たせることがポイントです。

また、3サブセクションを結合した本文が下限2,700文字を満たしていても、Step 04 Reviewの
推敲で2,679文字へ短縮され、21文字不足で全レビューが不合格になる事例もありました。下限850文字
なら結合後下限は2,550文字となるため、生成目標約3,600文字を変えずに推敲による軽微な圧縮を
許容できます。

## 調整の目安

会話をさらに増やしたい場合は、まず `min_dialogue_blocks` を少しずつ増やします。

```json
{
  "min_dialogue_blocks": 24,
  "max_dialogue_blocks": 48
}
```

一度に大きく増やすと、短すぎる相づちや不自然な会話分割が増える可能性があります。
生成結果を確認しながら4〜8件程度ずつ変更するのが安全です。

説明や情景描写も増やしたい場合は、セリフ数を維持したまま `max_characters` を増やします。
`min_characters` を先に増やすと、有用な短めの出力を拒否しやすくなるため注意してください。

登場人物が1人だけのセクションでは、20件以上の発言が独白の細分化になる可能性があります。
その場合はアウトライン側で参加者を増やすか、セクションごとに件数を変えられる仕組みを
追加することを検討します。

## 検証失敗の調べ方

最終エラーには、具体的な失敗理由、試行回数、再試行フェーズ、状態ファイル、
トレースログのパスが表示されます。

品質検証に失敗したセクションは、次の場所へ保存されます。

```text
output/<run-id>/artifacts/sections/chapter-NNN-section-NNN.rejected.json
```

このファイルの `metrics` で次を確認できます。

- `non_whitespace_characters`: 空白を除いた本文文字数
- `dialogue_blocks`: セリフ数
- `narration_blocks`: ナレーション数
- `total_blocks`: 全ブロック数

`required` には実行時の上下限が保存されます。例えばセリフ数を満たしているのに
文字数だけ不足している場合は、プロンプトの再試行よりも文字数下限の見直しが有効です。

## 再生成とチェックポイント

生成条件を変更すると、レンダリング済みプロンプトのハッシュも変わります。
既存チェックポイントのハッシュが一致しない場合、そのセクションは新しい条件で再生成されます。

```powershell
python run_pipeline.py `
  --config examples/pipeline.openai.config.json `
  --run-id openai-scenario-001 `
  --from-step step-04-generate-sections `
  --force
```

途中で失敗しても、同じプロンプトハッシュで検証済みのチェックポイントは再利用されます。
これにより、成功済みセクションのAPI生成を繰り返さず、失敗したセクションから再開できます。

### キャラクター設定と画像を維持して本文だけ再生成する

キャラクタープロフィール、基準画像、表情画像、章節構成を維持したまま本文だけを
修正したい場合は、同じrun IDを指定してStep 04から再実行します。

```powershell
python run_pipeline.py `
  --config examples/pipeline.openai.config.json `
  --input examples/input.json `
  --run-id my-openai-scenario `
  --from-step step-04-generate-sections `
  --force
```

この実行では、Step 01〜03の成果物を既存runから読み込みます。そのため、次の内容は
再生成されません。

- キャラクターID、名前、役割、性格、話し方、外見
- キャラクターの基準画像
- 4×4表情シートと表情別画像
- 章数、節数、章タイトル、節タイトル、主要イベント

Step 04以降の本文、セリフ表情タグ、HTMLは更新されます。`--force`を付けない場合、
既存の節チェックポイントが再利用される可能性があります。本文を確実に作り直すときは
`--force`を指定します。Step 04より前の画像生成ステップは再開対象外なので、
`--force`を付けても既存画像は再生成されません。

### 分量を増やして再生成する

既存のOpenAI用設定を直接変更せず、用途別の設定ファイルへコピーして調整すると、
元の生成条件と比較しやすくなります。例えば本文と会話の両方を増やす場合は、次のように
設定します。

```json
{
  "scenario_body_generation": {
    "subsections_per_section": 3,
    "target_characters": 1200,
    "min_characters": 850,
    "max_characters": 1600,
    "min_dialogue_blocks": 10,
    "max_dialogue_blocks": 20,
    "require_event_mentions": true
  }
}
```

本文の密度を上げたい場合でも、最初からすべての値を大幅に増やさないことが重要です。

- 会話の往復を増やす: `min_dialogue_blocks`を4〜8件ずつ増やす
- 情景描写や説明の余地を増やす: `max_characters`を増やす
- 極端に短い出力を防ぐ: 結果を確認してから`min_characters`を増やす
- 長い独白を避ける: セリフ数と文字数を同時に確認する

下限を急に高くすると、自然なシナリオまで不合格になり、APIの再試行回数と料金が
増える場合があります。まず1章分を生成し、`rejected.json`の実測値と完成HTMLを
確認してから全体へ適用します。

文字数下限だけを満たさなかった場合、Step 04は生成済み本文を捨てずに補完生成を1回
行います。不足文字数をモデルへ伝え、既存の事実・出来事の順序・話者・会話の意図を
維持しながら、反応、行動、因果関係、次への遷移を加筆します。補完結果も通常と同じ
スキーマ、文字数、セリフ数、イベント網羅条件で再検証されます。

### 本文で追加された情報の引き継ぎ

各サブセクションの出力には `state_updates` が含まれます。本文中で追加・変更された永続的な
情報を、次のサブセクションへ渡す累積状態へマージします。

- `character_locations`: 変更されたキャラクターの現在地
- `possessions`: 変更されたキャラクターの現在の所持品一覧
- `known_information`: 後続でも有効な判明事項
- `relationship_changes`: 後続へ影響する関係変化
- `introduced_entities`: 新しく確立した人物、場所、組織、物、概念
- `unresolved_plot_threads`: 新しく残った疑問や伏線
- `resolved_plot_threads`: この場面で解決した既存の伏線
- `continuity_summary`: 次の場面が必要とする直近状況の短い要約

過去の本文全文は次のプロンプトへ連結しません。累積した構造化情報と直近要約を渡すため、
長いシナリオでもプロンプトの増加を抑えつつ、追加設定を後続へ維持できます。

### シナリオ修正時の再開位置を選ぶ

修正内容によって再開するステップを変えます。

| 修正したい内容 | 再開位置 | 維持される主な成果物 |
| --- | --- | --- |
| 本文の表現、会話量、描写量 | Step 04 | キャラクター設定、画像、アウトライン |
| 章節構成、主要イベント、参加人物 | Step 02 | 入力に記載したキャラクター概要 |
| キャラクター設定そのもの | Step 01 | なし。画像も新設定に合わせて再生成対象 |

`examples/input.json`のテーマや前提だけを変更し、古いアウトラインを維持したまま
Step 04から再開すると、本文と章節設計が食い違う可能性があります。前提、章構成、
主要イベントまで変える場合はStep 02から再生成します。

画像を維持しながらアウトラインだけ変えたい場合は、画像プロンプトとキャラクター設定を
変更せず、画像チェックポイントを再利用できる状態にします。ただし参加人物や設定が変化した
場合は、古い画像を無理に流用せずStep 03も再生成する方が安全です。

## キャラクター画像生成のノウハウ

### 表情ごとの個別生成より4×4表情シートを優先する

当初は、基準画像を生成した後に各表情を1枚ずつAPIで生成していました。この方式には次の問題があります。

- 表情の数だけAPI呼び出し、待ち時間、料金が増える
- 呼び出しごとに顔、髪型、衣装、構図が変化しやすい
- 途中失敗時に多数のAPIリクエストを再開管理する必要がある

現在はキャラクターごとに、基準画像1枚と4列×4行の表情シート1枚を生成します。API呼び出しは合計2回で、表情シートから16枚をローカルで切り出します。

```text
基準画像を生成
    ↓ 参照画像として使用
4×4表情シートを生成
    ↓ 固定座標で16等分
表情別画像を保存
```

固定順序は次のとおりです。

| 行 | 左から右への表情 |
| --- | --- |
| 1 | neutral, smile, serious, thinking |
| 2 | surprised, worried, confused, angry |
| 3 | sad, relieved, embarrassed, nervous |
| 4 | confident, doubtful, shocked, determined |

プロンプトでは同一人物、同一衣装、同一アングル、同一サイズを強く指定し、文字、ラベル、罫線、背景装飾を禁止します。固定座標で切り出すため、人物がセルをまたがないことと、セル内に余白を確保することも重要です。

既定の表情シートは2048×2048で、切り出し後は1枚512×512です。シートの幅と高さはそれぞれ4で割り切れる必要があります。

### 高品質な画像生成では短いタイムアウトを避ける

`gpt-image-2`で`quality: high`を使用すると、画像生成が120秒を超える場合があります。正常に処理中でもタイムアウトする可能性があったため、OpenAI用設定では300秒にしています。

```json
{
  "image_generation": {
    "quality": "high",
    "timeout_seconds": 300
  }
}
```

タイムアウト値を大きくすれば必ず成功するわけではありません。API混雑、ネットワーク、VPN、プロキシ、ファイアウォールも確認します。動作確認を優先する場合は、一時的に`quality`を`medium`へ下げる方法もあります。

### SDKとパイプラインの多重リトライに注意する

OpenAI SDKはAPI呼び出しを内部でリトライする場合があります。さらにパイプラインも`short_retry`、`prompt_revision`、`fallback`を実行するため、両方が有効だと終了まで数十分かかる可能性があります。

調査時は`trace.jsonl`の次のイベントを確認します。

- `step_started`: 現在の試行番号とリトライ段階
- `step_failed`: 失敗理由と1試行の所要時間
- `step_retry_scheduled`: 次に予定されたリトライ段階
- `image_generated`: APIまたはmockによる画像生成完了
- `image_checkpoint_loaded`: APIを呼ばず既存画像を再利用

短時間で原因を確認したい場合は、検証用設定でパイプライン側のリトライを一時的に減らします。

```json
{
  "retry_strategy": {
    "short_retries": 0,
    "prompt_revision_retries": 0,
    "fallback_enabled": false
  }
}
```

### 生成画像と切り出し画像の両方にチェックポイントを持たせる

基準画像、4×4表情シート、切り出した16枚のそれぞれにチェックポイントを保存します。切り出し画像だけが欠損・破損した場合はAPIを呼ばず、保存済み表情シートからその画像だけを復元できます。表情シートが欠損した場合も、基準画像を再利用して表情シートから再開できます。

```powershell
python run_pipeline.py `
  --config examples/pipeline.openai.config.json `
  --run-id openai-scenario-001 `
  --from-step step-03-generate-character-images
```

実行中のPythonプロセスが残っている状態で同じrun IDを二重実行しないでください。チェックポイントやrun-stateを同時更新する可能性があります。

### APIキーの環境変数名と実値を区別する

画像生成設定の`api_key_env`はAPIキー自体ではなく、キーを格納した環境変数名です。既定値は`OPENAI_API_KEY`です。

```json
"api_key_env": "OPENAI_API_KEY"
```

```powershell
$env:OPENAI_API_KEY = "your-api-key"
```

設定ファイルに`your-api-key`のようなプレースホルダーを実キーとして書くと、401 `invalid_api_key`になります。テキストと画像で同じOpenAIプロジェクトと権限を使う場合、キーを分ける必要はありません。権限分離や利用量管理が必要な場合に別キーを検討します。

### 旧runとの互換性に注意する

4×4方式では`available_expressions`を固定16表情として扱います。旧方式の`neutral`, `happy`, `sad`などを持つrunは、画像ステップだけをそのまま再開できません。新しいrun IDを使用して`step-01-generate-character-profiles`から実行してください。
